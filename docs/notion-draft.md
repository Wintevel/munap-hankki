# 문앞한끼

## 1. 프로젝트 개요

### 1.1 서비스 소개

문앞한끼는 대화나 대면이 부담스러운 사람이 전화, 초인종, 문 앞 전달 방식을 먼저 확인한 뒤 식사를 주문할 수 있도록 돕는 배달 서비스이다.

사용자는 메뉴를 보기 전에 안심 주문 설정을 확인한다. 전화 없음, 벨 없음, 문 앞 전달은 기본으로 활성화되어 있으며, 품절이나 주소 확인 같은 예외 상황은 전화 대신 앱 알림으로 안내받는다.

### 1.2 핵심 문구

> 내가 정한 방식 그대로 도착하는 한 끼.

### 1.3 프로젝트 목표

이 프로젝트의 목표는 배달 과정에서 발생할 수 있는 예고되지 않은 전화, 초인종 소리, 문 앞 대면의 부담을 줄이는 것이다.

기획 과제에서는 Figma로 `1440px` 데스크톱 웹 메인 화면을 설계한다. 프론트엔드 과제에서는 Vite와 React를 이용하여 안심 주문 설정, 메뉴 선택, 옵션과 수량 변경, 주문 확인, 상태 변화, 주문 완료까지 이어지는 흐름을 구현한다.

### 1.4 확정된 방향

| 항목 | 내용 |

### 1.5 구현 범위 원칙

- 먼저 과제의 필수 결과물만 완성한다.
- 선택 과제는 필수 결과물 완성 후 남은 시간과 난이도를 검토한 뒤 별도로 결정한다.
- API, HTTP 메서드, JSON, SQLite는 서버 연결을 가정한 개념 설계로만 작성한다.
- 실제 서버, 데이터베이스, 결제, 알림 발송은 구현하지 않는다.
- 직접 설명할 수 있도록 React Router나 전역 상태 라이브러리 없이 `useState`와 조건부 렌더링을 중심으로 구성한다.
- 구현하지 않은 기능은 최종 결과물에서 구현된 기능으로 적지 않는다.

## 2. 상황과 조사 배경

### 2.1 상황

대화나 대면이 부담스러운 사람은 식사가 필요해도 배달 과정에서 발생할 수 있는 전화, 초인종 소리, 문 앞 접촉 때문에 주문을 망설일 수 있다.

온라인에서 메뉴를 선택할 수 있더라도 주소 확인, 품절, 전달 과정에서 예고되지 않은 연락이 발생할 수 있다는 불안은 주문을 시작하거나 완료하는 데 방해가 될 수 있다.

문앞한끼는 메뉴보다 안심 주문 설정을 먼저 보여 준다. 사용자가 확인한 설정은 메뉴 선택, 주문 확인, 주문 완료 화면에도 반복하여 표시한다. 이를 통해 선택한 방식이 주문 과정에서 유지되고 있음을 확인할 수 있게 한다.

### 2.2 조사 근거

보건복지부의 2023년 고립·은둔 청년 실태조사에서는 응답자의 73.2%가 온라인 매체를 통해 외부 정보를 인지한다고 답했다. 타인의 시선에 대한 두려움은 62.0%, 대면 접촉에 대한 두려움은 47.8%, 식사가 불규칙하다는 응답은 72.4%였다.

이 결과는 온라인 기반 정보 접근과 대면 부담을 낮춘 서비스의 필요성을 검토할 수 있는 근거가 된다. 다만 온라인 정보 이용 비율이 온라인 음식 주문 선호를 직접 의미하는 것은 아니다. 또한 자발적으로 참여자를 모집한 조사이므로 결과를 모든 고립·은둔 청년에게 일반화하지 않는다.

화면에서는 사용자를 히키코모리라고 직접 규정하지 않고, 대화나 대면이 부담스러운 사람이라는 상황 중심의 표현을 사용한다.

### 2.3 접근성과 사용성 자료에서 얻은 원칙

W3C의 인지 접근성 자료와 GOV.UK 및 NHS의 서비스 패턴을 다음과 같이 적용한다.

- 한 단계에서 하나의 주요 행동에 집중한다.
- 현재 단계와 다음 행동을 명확하게 표시한다.
- 전체 주문 경로를 짧게 유지한다.
- 사용자가 선택한 설정을 주문 확정 전에 다시 보여 준다.
- 메뉴 가격, 옵션 추가 비용, 총액을 미리 표시한다.
- 전화 이외의 도움과 예외 연락 수단을 제공한다.
- 선택 상태를 색상만으로 표현하지 않는다.
- 뒤로 이동해도 기존 선택 내용을 유지한다.
- 주문 완료 여부와 다음에 일어날 일을 분명하게 안내한다.

참고한 기본 흐름은 다음과 같다.

`설정 확인 → 메뉴와 옵션 선택 → 주문 내용 확인 → 주문 완료`

### 2.4 조사 결과와 기획 판단의 구분

| 구분 | 내용 |
|---|---|
| 조사에서 확인한 내용 | 온라인 정보 접근 비율, 타인의 시선과 대면 접촉 부담, 불규칙한 식사 비율 |
| 접근성 자료에서 확인한 내용 | 짧은 경로, 명확한 단계, 답변 확인, 완료 상태 안내의 필요성 |
| 직접 판단한 내용 | 메뉴보다 안심 주문 설정을 먼저 배치 |
| 직접 판단한 내용 | 전화 대신 앱 알림으로 예외 상황을 전달 |
| 직접 판단한 내용 | 안심 설정 UI 카드를 메인 화면의 핵심 비주얼로 사용 |

## 3. 주요 사용자와 문제 정의

### 3.1 주요 사용자

문앞한끼의 주요 사용자는 대화나 대면이 부담스러운 사람이다.

- 집 밖에 나가 식사를 구하기 어려운 사람
- 전화 통화나 즉시 답변해야 하는 대화를 부담스러워하는 사람
- 배달 담당자와 문 앞에서 마주치는 상황을 피하고 싶은 사람
- 초인종 소리나 예고되지 않은 방문에 긴장하는 사람
- 요청한 전달 조건이 적용되었는지 반복해서 걱정하는 사람
- 많은 선택지와 복잡한 주문 단계 때문에 주문을 중단하는 사람

### 3.2 문제 정의

대화나 대면이 부담스러운 사용자는 음식이 필요해도 배달 과정에서 원하지 않는 전화, 초인종, 문 앞 접촉이 발생할 수 있다는 불안 때문에 주문을 시작하거나 완료하기 어렵다.

### 3.3 해결하려는 핵심 문제

연락과 전달 방식이 주문 과정의 마지막에 있거나 단순 요청 사항으로만 처리되면 사용자는 메뉴를 고르는 동안에도 원하지 않는 연락이 발생할 가능성을 걱정하게 된다. 문앞한끼는 연락과 전달 방식을 메뉴 선택 이전에 확인하게 하여 이 불확실성을 줄인다.

## 4. 서비스 가치와 핵심 기능

### 4.1 핵심 가치

문앞한끼는 사용자가 원하는 연락과 전달 방식을 메뉴 선택 전에 확인하게 하고, 그 설정이 주문 완료까지 유지되는 과정을 보여 준다.

### 4.2 문제와 해결 방식

| 사용자가 느낄 수 있는 부담 | 해결 방식 |
|---|---|
| 주문 후 전화가 올 수 있다는 불안 | 전화 없음 설정을 기본 활성화 |
| 배달 담당자와 마주칠 수 있다는 부담 | 문 앞 전달을 기본 활성화 |
| 초인종 소리에 대한 긴장 | 벨 없음 설정을 기본 활성화 |
| 예외 상황에서 전화가 올 수 있다는 걱정 | 앱 알림으로 안내 |
| 설정 적용 여부를 확신하기 어려움 | 메뉴, 주문 확인, 완료 화면에 설정을 반복 표시 |
| 복잡한 과정에서 주문을 중단함 | 짧은 단계와 명확한 진행 상태 제공 |
| 옵션과 추가 비용을 놓침 | 주문 확정 전에 옵션과 총액을 다시 표시 |

### 4.3 안심 주문 설정

메인 화면 오른쪽에 실제 설정 UI 카드 형태로 표시한다.

- 전화 없음
- 벨 없음
- 문 앞 전달
- 예외 연락은 앱 알림

각 설정은 색상, 아이콘, 체크 표시, 상태 문구를 함께 사용해 활성화 여부를 전달한다.

세 설정은 하나의 통합 모드가 아니라 각각 독립된 토글로 구현한다.

### 4.4 메뉴와 옵션

메뉴는 Mock JSON 또는 JavaScript 배열로 표시한다. 사용자는 메뉴 옵션과 수량을 변경하고 장바구니에 추가할 수 있다. 선택 결과와 총액은 주문 요약에 즉시 반영한다.

### 4.5 주문 상태와 앱 알림

주문 과정은 설정 확인, 메뉴 선택, 주문 확인, 처리 중, 주문 완료 상태로 나눈다. 품절이나 주소 확인 같은 예외 상황은 실제 알림 전송 대신 화면의 앱 알림 카드로 표현한다.

### 4.6 구현 범위

이번 프로젝트에서 구현하는 기능은 다음과 같다.

- 안심 주문 설정 확인과 변경
- Mock 메뉴 목록 표시
- 메뉴 옵션 선택
- 수량 변경
- 장바구니 또는 주문 요약 갱신
- 주문 내용 확인
- 주문 처리 상태 변화
- 주문 완료 화면
- 예외 상황 앱 알림 표현

실제 결제, 위치 추적, 음식점 관리, 배달 담당자 배정, 실제 전화 차단, 실제 알림 발송, 실제 서버와 데이터베이스 연결은 구현하지 않는다.

## 5. 예상 전체 이용 순서

1. 사용자가 문앞한끼 메인 화면에 들어온다.
2. 왼쪽 소개 영역에서 서비스 대상과 해결 문제를 확인한다.
3. 오른쪽 카드에서 전화 없음, 벨 없음, 문 앞 전달 상태를 확인한다.
4. 필요하면 각 설정을 변경한다.
5. 예외 연락 방식이 앱 알림인지 확인한다.
6. 안심 설정 확인하고 메뉴 보기 버튼을 누른다.
7. Mock 데이터로 표시된 메뉴를 살펴본다.
8. 메뉴 옵션과 수량을 선택한다.
9. 메뉴를 장바구니에 추가한다.
10. 주문 요약에서 메뉴, 옵션, 수량, 가격을 확인한다.
11. 안심 주문 설정이 유지되고 있는지 다시 확인한다.
12. 주문 버튼을 누른다.
13. 주문 처리 중 상태를 확인한다.
14. 주문 완료 화면에서 주문 결과와 안심 설정을 확인한다.

## 6. Figma 메인 화면 기획

### 6.1 화면 목적

메인 화면만 보고 다음 내용을 파악할 수 있어야 한다.

- 누구를 위한 서비스인지
- 어떤 문제를 해결하는지
- 핵심 기능이 무엇인지
- 다음에 무엇을 해야 하는지

### 6.2 화면 구조

- 화면 유형: 데스크톱 웹
- 프레임 너비: `1440px`
- 최대 콘텐츠 너비: `1200px`
- 기본 그리드: 12열
- 왼쪽: 서비스 소개와 주요 버튼
- 오른쪽: 실제 안심 주문 설정 UI 카드
- 주요 색상: `#2E9B83`
- 밝은 배경: `#EAF8F4`
- CTA 글자: `#17211F`
- 카드 배경: `#FFFFFF`
- 서체: Noto Sans KR의 Regular, Medium, Bold
- 아이콘: 둥근 선형 아이콘

### 6.3 왼쪽 소개 영역

- 서비스명: 문앞한끼
- 사용자 설명: 대화나 대면이 부담스러운 사람을 위한 배달 서비스
- 핵심 문구: 내가 정한 방식 그대로 도착하는 한 끼.
- 보조 설명: 전화, 벨, 문 앞 전달 방식을 메뉴를 고르기 전에 먼저 확인할 수 있다는 내용
- 이용 순서: 안심 설정, 메뉴 선택, 주문 확인
- 다음 행동: 오른쪽 설정 카드의 안심 설정 확인하고 메뉴 보기 버튼

`#2E9B83` 버튼에는 `#17211F` 글자를 사용한다. 두 색상의 명암 대비는 약 `4.81:1`로 일반 크기 글자의 접근성 기준을 충족한다.

### 6.4 오른쪽 설정 카드

| 설정 | 기본 상태 |
|---|---|
| 전화 없음 | 활성화 |
| 벨 없음 | 활성화 |
| 문 앞 전달 | 활성화 |
| 예외 연락 | 앱 알림 |

설정 카드는 장식용 이미지가 아니라 React에서 실제로 변경할 수 있는 핵심 기능으로 연결한다.

카드 하단에는 안심 설정 확인하고 메뉴 보기 버튼을 배치하여 다음 행동을 명확히 안내한다.

### 6.5 실제 Figma 제작 결과

- `1440px × 900px` 크기의 `Main/Desktop 1440` 프레임을 제작했다.
- 왼쪽에는 대상 사용자, 해결 문제, 서비스 가치, 이용 순서를 배치했다.
- 오른쪽에는 전화 연락, 초인종, 문 앞 전달 설정과 앱 알림 안내 및 CTA를 배치했다.
- 색상 변수 18개, Noto Sans KR 텍스트 스타일 5개, 그림자 스타일 1개를 별도로 정리했다.
- 아이콘은 각 설정의 의미를 보조하는 둥근 선형 형태로 통일했다.
- 현재 Figma 결과물은 정적 메인 화면이다. 토글 변경과 메뉴 단계 이동 Prototype은 연결하지 않았다.
- 화면 안의 버튼과 토글은 시각 요소로 배치했으며 재사용 가능한 Figma 컴포넌트와 Variant로 만들지는 않았다.
- 실제 토글 변경과 화면 이동은 React에서 구현한다.

### 7.3 `useState`로 관리할 정보

| 상태 | 실제 저장 정보 | 초기값 | 변경 시점 |
|---|---|---|---|
| `currentStep` | 현재 화면 단계 | `setup` | 다음, 이전, 주문, 처음부터 다시 보기 버튼 클릭과 처리 완료 |
| `safeOrderSettings` | 전화 없음, 벨 없음, 문 앞 전달, 예외 연락 방식 | 세 설정 활성화, 앱 알림 | 각 토글 클릭 |
| `selectedMenuId` | 현재 선택한 메뉴의 ID | 첫 번째 메뉴 ID | 메뉴 카드 클릭 |
| `selectedOptions` | 양과 음료의 선택값 | 기본 양, 음료 선택 안 함 | 옵션 라디오 버튼 선택 |
| `quantity` | 선택 수량 | `1` | 증가와 감소 버튼 클릭 |
| `cart` | 추가한 메뉴의 스냅샷, 옵션, 수량, 단가 | 빈 배열 | 메뉴 추가와 항목 삭제 |
| `notificationConfirmed` | 상세 주소 확인 앱 알림의 확인 여부 | `false` | 완료 화면의 확인 버튼 클릭 |

안심 주문 설정의 초기 상태 예시는 다음과 같다.

```js
{
  noPhoneCall: true,
  noDoorbell: true,
  leaveAtDoor: true,
  exceptionContact: 'app-notification'
}
```

메뉴 원본 목록과 공통 옵션은 변경하지 않는 Mock JavaScript 배열이므로 상태로 만들지 않았다. 선택 메뉴, 켜진 설정 수, 장바구니 수량, 총액, 선택 옵션의 추가 금액은 기존 상태에서 계산하며 별도 상태로 중복 저장하지 않았다. 주문 처리 중 여부도 별도의 `orderStatus`를 만들지 않고 `currentStep`의 `processing` 값으로 표현했다.

## 9. SQLite에 저장할 정보

| 빌드 오류 | Vite 프로젝트와 패키지 설치 명령 실행 | 샌드박스가 `esbuild` 하위 프로세스를 막아 발생한 `spawn EPERM`을 확인하고, 빌드 명령에 한정된 실행 권한으로 다시 빌드 | 코드 오류와 실행 환경 권한 문제를 구분하고 프로덕션 산출물을 검증하기 위해 |
