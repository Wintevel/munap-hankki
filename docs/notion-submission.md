# 한눈에 보기

문앞한끼는 전화 주문, 대면 전달, 초인종 소리가 부담스러운 사용자가 원하는 배달 방식을 먼저 정한 뒤 식사를 주문할 수 있도록 기획한 데스크톱 웹 서비스다.

- 주요 사용자: 대화나 대면이 부담스러운 사람
- 해결 문제: 전화 연락, 벨 소리, 문 앞 대면에 대한 부담
- 핵심 기능: 전화하지 않기, 벨 누르지 않기, 문 앞에 두기를 메뉴 선택 전에 설정
- 예외 연락: 전화 대신 앱 알림
- 핵심 문구: 내가 정한 방식 그대로 도착하는 한 끼.

# 공부한 내용

Figma에서는 프레임, 레이어, 오토레이아웃, 컴포넌트, Variant, Prototype의 역할을 공부했다. 와이어프레임은 장식보다 정보의 우선순위와 사용자의 이동 순서를 확인하는 초안이라는 점을 이해했다. 이번 결과물에서는 대상 사용자, 해결 문제, 핵심 기능, 다음 행동이 한 화면에서 드러나도록 구성했다.

React에서는 JSX를 반환하는 함수 컴포넌트, props, `useState`, 이벤트 처리, 배열의 `map()`을 이용한 목록 렌더링, 조건부 렌더링을 적용했다. 상태를 바꾸면 React가 해당 상태를 사용하는 화면을 다시 렌더링한다는 흐름을 메뉴 선택과 장바구니에 연결했다.

# 직접 해본 것

## Figma 기획 화면

`1440×900` 데스크톱 메인 화면을 제작했다. 왼쪽에는 서비스 소개와 3단계 이용 순서를, 오른쪽에는 실제로 조작할 안심 설정 카드를 배치했다. 주요 색상은 `#2E9B83`, 연한 배경은 `#EAF8F4`, CTA 글자는 `#17211F`를 사용했다. 음식 사진이나 기존 배달 앱의 목록 구조를 중심 비주얼로 사용하지 않았다.

현재 Figma 결과물은 정적 메인 화면이다. Prototype, 재사용 가능한 Button과 SettingToggle 컴포넌트, Variant는 제작하지 않았다.

## React 구현

Mock JavaScript 배열로 서로 다른 메뉴 4개를 표시했다.

- 간장 닭 덮밥
- 바질 토마토 파스타
- 닭가슴살 샐러드
- 단호박 수프 세트

구현한 사용자 흐름은 다음과 같다.

1. 세 안심 설정을 확인하거나 각각 변경한다.
2. 메뉴 4개 중 하나를 고른다.
3. 양과 음료 옵션, 수량을 선택한다.
4. 장바구니에 추가하고 총액을 확인한다.
5. 주문 확인 화면에서 메뉴와 안심 설정을 다시 확인한다.
6. 주문 버튼을 누르면 약 `1.5초`의 처리 중 화면이 나타난다.
7. 주문 완료 화면에서 주문 요약과 안심 설정을 확인한다.
8. 상세 주소 확인이 필요한 예외 상황에서는 전화 대신 앱 알림을 확인한다.

같은 메뉴와 같은 옵션을 다시 추가하면 수량을 합산한다. 주문 확인 화면에서 마지막 항목을 삭제하면 빈 확인 화면에 남지 않고 메뉴 선택 화면으로 돌아간다. 실제 결제와 주문 전송은 발생하지 않는다.

# 컴포넌트 분류

| 컴포넌트 | 역할 |
|---|---|
| `App` | 공통 상태 관리와 화면 단계 연결 |
| `Brand` | 서비스명과 아이콘 표시 |
| `MenuIcon` | 메뉴별 색상과 선형 아이콘 표시 |
| `SafeToggle` | 안심 설정 한 항목과 독립 토글 표시 |
| `SetupScreen` | 서비스 소개와 안심 설정 메인 화면 |
| `ProgressHeader` | 주문 진행 단계 표시 |
| `MenuScreen` | 메뉴, 옵션, 수량, 장바구니 요약 표시 |
| `SafetySummary` | 현재 안심 설정과 앱 알림 방식 요약 |
| `ReviewScreen` | 주문 항목, 총액, 안심 설정 최종 확인 |
| `ProcessingScreen` | 주문 처리 중 상태 표시 |
| `CompleteScreen` | 주문 완료와 주소 확인 앱 알림 표시 |

화면 단위 컴포넌트와 반복되는 작은 UI만 분리해 구조를 복잡하게 만들지 않았다.

# useState로 관리한 정보

| 상태 | 저장 정보 |
|---|---|
| `currentStep` | `setup`, `menu`, `review`, `processing`, `complete` 중 현재 단계 |
| `safeOrderSettings` | 전화 없음, 벨 없음, 문 앞 전달, 예외 앱 알림 |
| `selectedMenuId` | 현재 선택한 메뉴 ID |
| `selectedOptions` | 양과 음료의 선택값 |
| `quantity` | 선택 수량 |
| `cart` | 메뉴, 옵션, 수량, 단가가 담긴 장바구니 |
| `notificationConfirmed` | 상세 주소 확인 앱 알림의 확인 여부 |

메뉴 원본과 공통 옵션은 변경되지 않는 Mock 배열로 두었다. 총액과 전체 수량은 `cart`에서 계산하고, 선택 메뉴와 옵션 추가 금액도 기존 값에서 계산해 중복 상태를 만들지 않았다.

# Figma와 실제 구현에서 달라진 부분

- Figma는 서비스의 대상, 문제, 핵심 기능, CTA가 드러나는 첫 화면만 제작했다.
- React에서는 첫 화면 디자인을 바탕으로 메뉴 선택, 주문 확인, 처리 중, 완료 화면을 추가했다.
- Figma의 토글은 정적이지만 React의 세 토글은 각각 독립적으로 변경된다.
- React 메뉴는 음식 사진 대신 종류별 색상과 둥근 선형 아이콘으로 구분했다.
- Figma Prototype은 만들지 않았고 화면 연결은 React의 `currentStep`과 조건부 렌더링으로 구현했다.

# 서버 연결을 가정한 개념 설계

이번 React 프로젝트에는 서버와 SQLite를 연결하지 않았다. 아래 내용은 실제 서비스로 확장한다고 가정한 설계다.

| HTTP 메서드 | API | 목적 |
|---|---|---|
| `GET` | `/api/restaurants` | 음식점 목록 조회 |
| `GET` | `/api/restaurants/:restaurantId/menus` | 메뉴와 옵션 조회 |
| `POST` | `/api/orders` | 메뉴와 안심 설정을 포함한 주문 생성 |
| `GET` | `/api/orders/:orderId` | 주문 내용과 상태 조회 |
| `PATCH` | `/api/orders/:orderId/preferences` | 주문 확정 전 안심 설정 변경 |
| `GET` | `/api/notifications` | 앱 알림 목록 조회 |
| `PATCH` | `/api/notifications/:notificationId/read` | 알림 읽음 처리 |

주문 요청 JSON의 간단한 예시는 다음과 같다.

```json
{
  "items": [
    {
      "menuId": "menu-001",
      "quantity": 2,
      "options": {
        "portion": "regular",
        "drink": "none"
      }
    }
  ],
  "deliverySettings": {
    "noPhoneCall": true,
    "noDoorbell": true,
    "leaveAtDoor": true,
    "exceptionContact": "app-notification"
  }
}
```

SQLite에는 `restaurants`, `menus`, `menu_options`, `orders`, `order_items`, `order_item_options`, `order_delivery_settings`, `notifications` 정보를 저장한다고 가정했다.

# 해결한 오류와 배운 점

- Vite 빌드 중 실행 환경이 `esbuild` 하위 프로세스를 막아 `spawn EPERM`이 발생했다. 코드 오류가 아니라 권한 문제임을 확인하고 빌드 명령에 필요한 권한으로 다시 실행해 프로덕션 빌드에 성공했다.
- 처리 중 화면인데 진행 단계가 주문 완료로 표시되는 문제를 발견해 해당 단계 문구를 `접수 중`으로 수정했다.
- 주문 확인 화면에서 마지막 항목을 삭제하면 빈 화면에 남던 문제를 발견해 메뉴 선택 화면으로 돌아가도록 수정했다.
- 총액처럼 기존 상태에서 계산할 수 있는 값은 별도의 상태로 저장하지 않는 편이 상태 불일치를 줄인다는 점을 배웠다.
